# 코드 분석 프롬프트 (React/TS 예시)

프로젝트 레포에서 Claude Code로 실행하면, 코딩 스타일을 자동 분석하여 coding-profile.md를 생성합니다.

> **이 파일은 React/TypeScript 프로젝트용 예시입니다.**
> 다른 스택(Python, Go, Vue 등)에서는 `/setup`의 코딩 프로필 생성 기능이
> 스택에 맞는 분석 카테고리를 동적으로 생성합니다.

---

```bash
cd ~/your-company-project
claude --plugin-dir ~/Desktop/ai/code-forge
```

아래 프롬프트를 복붙:

---

```
내 코딩 스타일을 분석해줘. 이 프로젝트의 코드를 읽고 아래 항목별로 "이 사람은 이렇게 쓴다"를 정리해줘.

추측하지 말고 실제 코드에서 발견한 패턴만 근거(파일:라인)와 함께.

## 분석 항목

### 1. 추상화 습관
- 같은 코드가 반복될 때 몇 번째에서 추출하는가? (2곳? 3곳?)
- 커스텀 훅을 얼마나 적극적으로 추출하는가?
- 훅 반환값이 보통 몇 개인가? 구조분해 vs 객체?
- 공통 컴포넌트를 어떤 수준으로 만드는가? (Flat? Compound? 도메인 래퍼?)

### 2. 상태 관리
- 서버 상태 관리 도구: React Query? SWR? 직접 fetch?
- 클라이언트 전역 상태: Jotai? Zustand? Redux? Context?
- 필터/폼 상태를 어디에 두는가? useState? URL? 전역?
- 서버 데이터를 자식에 어떻게 전달하는가? props? 캐시 직접 호출? Context?

### 3. 컴포넌트 설계
- 평균 컴포넌트 줄 수는? 가장 큰 컴포넌트는 몇 줄?
- 분리 기준이 보이는가? (줄 수? 역할? 반복?)
- children 패턴을 쓰는가? render props?
- 스타일 코드 위치: 인라인? styled? CSS module? 별도 파일?

### 4. 타입
- interface vs type 비율은?
- 타입 정의 위치: 인라인? 별도 파일? 서버 타입 공유?
- any 사용 빈도는?
- 제네릭 활용 수준은?

### 5. 에러/로딩 처리
- 로딩 상태: if문? Suspense? 커스텀 훅?
- 에러 처리: try-catch? 인터셉터? onError? ErrorBoundary?
- 빈 상태 처리를 하는가?

### 6. 이벤트 핸들러
- 인라인 vs handleXxx 비율은?
- 네이밍 패턴: handle? on? 혼용?

### 7. 폴더 구조
- 기능별? 레이어별? 페이지별?
- 공통 코드는 어디에?

### 8. 커밋/PR 스타일
- 커밋 메시지 패턴 (git log --oneline -20 확인)
- 커밋 크기: 큰 단위? 작은 단위?

### 9. 기타 패턴
- Early return 사용 빈도
- 옵셔널 체이닝 vs null 체크 스타일
- 비동기 패턴: async/await? .then()?
- import 정리 스타일

## 출력 형식

각 항목별로:
```
### {항목}
**패턴:** {발견된 패턴}
**근거:** {파일:라인 예시 2-3개}
**빈도:** {항상/대부분/가끔/드물게}

## 분석 결과 활용

분석 결과를 이 세션(code-forge)에 복붙하면,
기존 profile.md의 토론 결과와 통합하여 최종 프로필을 완성합니다.

### 통합 시 우선순위
1. 토론으로 확정된 기준 (Q1~Q12) — 이미 검증됨
2. 회사 코드에서 발견된 실제 패턴 — 습관 확인
3. 상충하면 → 사용자에게 "토론 결과는 이건데, 실제 코드는 저렇습니다. 어떤 걸 기준으로 할까요?" 물어보기

### 남은 질문 (아직 토론 안 한 것)
- Q13. window.confirm vs 커스텀 모달 — 과제에서의 판단
- 폴더 구조 선호 (기능별? 레이어별?)
- import 정리 스타일
- Early return 기준
- 비동기 패턴 선호
- 네이밍 컨벤션 (변수, 함수, 컴포넌트)
- 테스트 작성 스타일 (RTL? 단위? 통합?)
- git 브랜치 전략

이것들은 회사 코드 분석으로 자연스럽게 답이 나올 수 있음.
