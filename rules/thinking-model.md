# 작업 절차 (Working Protocol)

> 코드베이스가 스승이다 — 기존 코드에는 보이지 않는 이유가 있을 수 있다.
> 먼저 관찰하고 존중하되, 새 코드는 프로젝트 규칙 기준으로 작성한다.

---

## 불변 제약 (Invariants)

모든 코드 작업에 항상 적용:

1. **읽기 우선**: 수정 전 반드시 Read/Grep 도구를 실행한다. 기억에서 추론하지 않는다
2. **패턴 준수**: 관찰한 기존 패턴(구조, 네이밍, 스타일)을 따른다. 기존 코드에 안티패턴이 보이더라도 그 자리에 있는 이유(버그 우회, 레거시 호환 등)가 있을 수 있으므로 임의로 수정하지 않는다. 새로 작성하는 코드는 forbidden-patterns 기준을 따르되, 기존 안티패턴을 발견하면 사용자에게 알리고 개선 여부를 확인받는다
3. **정책 보존**: 비즈니스 로직(날짜 계산, 필터 조건, disabled 조건)을 임의 변경하지 않는다
4. **최소 변경**: 요청받은 것만 수정한다
5. **스코프 준수**: 작업 대상 앱/도메인 외 파일은 명시 요청 없이 수정하지 않는다

---

## 작업 루프

```
GROUND → APPLY → VERIFY
  ↑                 ↓
  └─── ADAPT ←──────┘
       (실패 시만)
```

### GROUND (맥락 확보) — 도구 실행 필수

- 대상 앱/도메인 스코프 확인
- 대상 파일 Read + 유사 구현 Grep (3개+ 독립 파일은 병렬 읽기)
- 관찰 결과로 규모 판단:

| 규모 | 기준 | 행동 |
|------|------|------|
| **S** | 1파일, 명확한 수정 | 즉시 APPLY |
| **M** | 2-5파일, 기존 패턴 적용 | 패턴 확인 후 APPLY |
| **L** | 5+파일 또는 새 설계 | architect/analyst로 설계 검토 → 단계적 APPLY |

- (버그 수정 시) 코드를 읽고 의도를 먼저 판단한다:
  - 코드에 주석/테스트/네이밍으로 의도가 명확 → 판단 후 진행
  - 의도를 코드에서 읽을 수 없음 → 사용자에게 확인 후 진행
  - MEDIUM 이상의 복잡도면 2-3가지 해결 옵션을 도출하고 사용자에게 제시한다
- (리팩토링 시) 정책 보호 테스트를 먼저 작성한다 (`/test` 활용)
- (외부 라이브러리 수정 시) `node_modules`에서 소스를 추적하여 내부 동작을 확인한다

- (복잡한 문제/설계 시) 가정을 분류하고 문제를 재정의한다:
  - **가정 분류**: A(물리적 제약 — 유지) / B(업계 관행 — 제거 가능) / C(미검증 — 검증 필요)
  - **문제 재정의**: "왜 이렇게 해야 하지?" → 근본 제약만 남기고 해결 경로 2-3개 도출
  - 기존 접근이 안 풀릴 때: debate(교차 검증) 또는 다각도 분석(역전, SCAMPER) 고려

### APPLY (실행)

GROUND에서 관찰한 패턴을 적용하여 구현한다. 불변 제약 5가지를 준수한다.

### VERIFY (검증) — 도구 실행 필수

```bash
yarn tsc --noEmit | head -50
yarn lint
```

- [ ] lint/tsc 통과? (실행 결과로 확인)
- [ ] 기존 동작 보존? (테스트가 있으면 실행, 없으면 코드를 읽고 영향받는 호출 경로를 추적하여 확인)
- [ ] 원래 요청 범위만 수정?
- [ ] (정책 변경 시) 테스트 작성?
- [ ] 수정 컴포넌트/함수의 모든 호출처를 grep으로 확인했는가?
- [ ] 배열/복합 데이터의 0건, 1건, N건 렌더링/처리를 확인했는가?
- [ ] UI 변경이면 dev 서버 또는 Playwright로 실제 화면을 확인했는가?
- [ ] 상태 캐시/쿼리 무효화 순서가 비동기 흐름과 맞는가?
- [ ] (커밋 시) 커밋 범위 한정, 미해결 리스크 PR 명시?
- [ ] (커밋 시) TC 없는 수정 파일이 있으면 `/test`로 생성 후 실행?
- [ ] (Stop 훅 경고 확인) TC 없음/정책 미갱신 경고에 대응했는가?

**통과 → 완료** / **실패 → ADAPT**

### ADAPT (적응 — 실패 시에만)

검증 실패 시, 같은 실수를 반복하지 않기 위한 성찰 루프.

**진입 조건:**
- 기술적 실패 (lint/tsc/build): 즉시 진입
- 사용자 거절/수정 요청: 의도 재확인 후 진입

**절차:**
1. **증상**: 무엇이 실패했는가?
2. **근인**: 왜 실패했는가? (표면이 아닌 근본 원인)
3. **교정**: 접근법을 어떻게 바꿀 것인가?
4. GROUND로 복귀 (교정된 맥락으로)

**자동 트리거:**
- `quality-gate.sh`가 실패를 감지하면 `.claude/state/reflect.flag`가 생성되고, 다음 세션 시작 시 `session-init.sh`가 이 파일을 감지하여 ADAPT를 강제한다. flag는 다음 quality-gate 통과 시 자동 삭제.
- 의도적으로 건너뛰려면 flag 본문에 `ack: <이유>` 추가 또는 `rm .claude/state/reflect.flag`.

**한계:**
- 같은 근본 원인으로 2회 실패 → 가정 재분류(A/B/C) + 문제 재정의 시도 + 사용자에게 상황 보고
- 3회 실패 → 작업 중단, 사용자 판단 요청

## 에이전트 연계

L 규모이거나 독립적인 전문 작업이 필요할 때 에이전트를 위임한다. 독립 작업은 병렬 호출한다.

| 단계 | 에이전트 |
|------|----------|
| GROUND | explore (패턴/구조 탐색) |
| APPLY (L) | architect (설계 검토), implementor (구현) |
| VERIFY | lint-fixer (오류 수정), assayer (테스트 생성) |

### 테스트 라우팅

| 사용자 의도 | 경로 |
|------------|------|
| "테스트 환경 세팅" | `/test --setup` |
| "이 파일 테스트 만들어" | `/test {파일경로}` |
| "E2E 만들어" | `/test {페이지경로} --e2e` |
| "전체 테스트 실행" | `/test --all` |
