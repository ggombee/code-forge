## 지금까지의 이야기

1편에서 나는 AI 코딩 에이전트의 사고 과정을 설계하기 시작했고, "프롬프트가 아니라 워크플로우를 설계해야 한다"는 인사이트를 얻었다. 2편에서는 거기서 더 나아가 Chain of Thought와 인지적 도제 이론을 결합한 **6단계 인지 흐름 모델**(READ → REACT → ANALYZE → RESTRUCTURE → STRUCTURE → REFLECT)을 설계했다. 에이전트에게 "어떤 순서로 생각할지"를 명시적으로 코드화한 것이다.

결과는 나쁘지 않았다. 정책 누락이 줄었고, 에이전트가 맥락 없이 코드를 생성하는 일이 확실히 감소했다. 14개 페르소나 에이전트, 5단계 품질 게이트, /start → /done 워크플로우까지. 48개 .md 파일, ~10,000줄의 에이전트 설정이 실제 프로젝트에서 잘 돌아가고 있었다.

하지만 문제가 있었다.

**6단계가 너무 많았다.**

## 1. 왜 6단계를 4단계로 줄였나

6단계 모델은 이론적으로 완결적이었다. 읽고(READ), 반응하고(REACT), 분석하고(ANALYZE), 재구조화하고(RESTRUCTURE), 구조화하고(STRUCTURE), 성찰한다(REFLECT). 인지 심리학 교과서를 옮긴 것 같은 깔끔한 모델.

실무에서의 문제는 세 가지였다.

**첫째, 토큰 낭비.** 6단계를 전부 밟으라고 시키면 에이전트가 각 단계에서 상당한 양의 추론 토큰을 소모한다. 간단한 버그 수정인데도 RESTRUCTURE 단계를 거치며 "현재 구조를 분석합니다..."를 출력하고 있었다. 이전 글에서 "100개 규칙보다 1개의 올바른 원칙이 더 강력하다"고 썼는데, 사고 모델도 마찬가지였다.

**둘째, 단계 간 경계가 모호했다.** READ와 REACT의 경계가 뭔가? RESTRUCTURE와 STRUCTURE는 뭐가 다른가? 나도 명확히 설명하기 어려웠는데, 에이전트가 이걸 구분할 수 있을 리 없었다. 모호한 단계를 두면 에이전트는 "어디에 속하는지 판단하느라" 추가 토큰을 쓴다.

**셋째, 실패 루프가 없었다.** 6단계 모델은 한 방향으로만 흐른다. 검증에서 실패하면? 처음부터 다시 돌아가야 하나, 아니면 REFLECT에서 끝내야 하나? 실패 시 어디로 돌아갈지가 정의되어 있지 않았다.

그래서 GROUND → APPLY → VERIFY + ADAPT로 바꿨다.

<!-- 📸 [사고 모델 비교 다이어그램] 왼쪽: READ→REACT→ANALYZE→RESTRUCTURE→STRUCTURE→REFLECT (6단계, 선형).
     오른쪽: GROUND→APPLY→VERIFY + 실패시 ADAPT→GROUND 피드백 루프 (4단계, 순환).
     두 모델의 구조적 차이가 한눈에 보이도록. 생성 이미지 권장. -->

```
GROUND → APPLY → VERIFY
  ↑                 ↓
  └─── ADAPT ←──────┘
       (실패 시만)
```

### GROUND (맥락 확보)

"코드베이스가 스승이다 — 기존 코드에는 보이지 않는 이유가 있을 수 있다."

수정 전에 반드시 Read/Grep 도구를 실행한다. 기억에서 추론하지 않는다. 유사 구현을 찾고, 관찰 결과로 작업 규모를 판단한다.

| 규모 | 기준 | 행동 |
|------|------|------|
| **S** | 1파일, 명확한 수정 | 즉시 APPLY |
| **M** | 2-5파일, 기존 패턴 적용 | 패턴 확인 후 APPLY |
| **L** | 5+파일 또는 새 설계 | architect/analyst로 설계 검토 후 단계적 APPLY |

### APPLY (실행)

GROUND에서 관찰한 패턴을 적용하여 구현한다. 5가지 불변 제약을 준수한다: 읽기 우선, 패턴 준수, 정책 보존, 최소 변경, 스코프 준수.

### VERIFY (검증)

lint/tsc 통과? 기존 동작 보존? 원래 요청 범위만 수정? 도구 실행으로 확인한다. 통과하면 완료.

### ADAPT (적응 - 실패 시만)

여기가 핵심이다. 6단계 모델에 없던 것.

VERIFY에서 실패하면 ADAPT로 진입한다. 증상은 무엇이고, 근인은 무엇이며, 접근법을 어떻게 바꿀 것인가. 그리고 **교정된 맥락으로** GROUND에 복귀한다. 같은 근본 원인으로 2회 실패하면 접근법을 전환하고, 3회 실패하면 작업을 중단하고 사용자에게 판단을 요청한다.

이 ADAPT 루프가 추가되면서 에이전트가 "같은 실수를 반복하지 않는" 구조가 생겼다. 6단계 모델에서는 실패하면 처음부터 다시 6단계를 밟아야 했는데, 이제는 실패 원인을 특정하고 교정된 상태로 복귀한다.

**결과적으로 4개 파일 수백 줄의 검증 규칙이 1개 파일 99줄로 압축됐다.** 오히려 에이전트가 더 잘 따랐다. 규칙이 많으면 "어떤 규칙을 우선할지" 판단하느라 토큰을 쓰고, 정작 중요한 규칙을 놓친다.

## 2. 특정 스택에서 범용 플러그인으로

사고 모델을 단순화하면서, 더 큰 문제가 보이기 시작했다. **이 시스템이 특정 스택에 묶여 있다는 것.**

코드에서 "이 코드 건드리면 터진다"는 말을 많이 듣는데, 에이전트 하네스에서도 똑같은 말이 나오기 시작했다. 프롬프트 한 줄 고쳤는데 다른 에이전트가 이상하게 동작한다. 스택 컨벤션을 바꾸고 싶은데 규칙 파일 곳곳에 하드코딩되어 있어서 어디를 건드려야 할지 모르겠다. 클린 아키텍처 2장에서 말하는 "경직된 시스템" 그 자체였다 — 작은 변경인데 비용이 크고, 어떤 변경은 사실상 불가능한 상태.

React + Next.js Pages Router + Jotai + Emotion 같은 스택 컨벤션이 규칙 파일 곳곳에 하드코딩되어 있었다. 새로운 프로젝트에서 같은 에이전트 분리 원칙이나 /start → /done 워크플로우를 쓰고 싶었는데, Jotai나 Emotion 컨벤션까지 따라오는 건 의미가 없었다.

결정적 계기는 Claude Code의 Plugin System이었다. `.claude-plugin/plugin.json` 매니페스트만 만들면 agents/, skills/, hooks/를 자동으로 감지해주는 구조. 이건 내가 원하던 것 — "도메인 로직은 빼고, 워크플로우 뼈대만 배포하는 것"이 가능해졌다. 실험하던 패턴들을 Claude Code 플러그인으로 패키징하면 `claude plugin add` 한 줄로 어떤 프로젝트에든 적용할 수 있다.

핵심 변경은 **모듈 시스템** 도입이었다:

```json
// presets/standard.json
{
  "modules": {
    "framework": "react-nextjs-pages",
    "state": "jotai-tanstack",
    "styling": "emotion",
    "testing": "jest"
  }
}
```

규칙 파일에 하드코딩되어 있던 스택 컨벤션을 `modules/` 디렉토리로 추출하고, 프리셋으로 조합할 수 있게 만들었다. `/setup` 한 번 실행하면 프로젝트 스택에 맞는 모듈이 자동으로 적용된다.

<!-- 📸 [모듈 시스템 before/after] 왼쪽: Jotai+Emotion이 에이전트 규칙에 박혀 있는 상태.
     오른쪽: modules/ 디렉토리에서 레고처럼 조립하는 구조.
     "특정 스택 전용 → 어떤 스택에도 적용 가능" 메시지. 생성 이미지 권장. -->

```
code-forge/
├── .claude-plugin/plugin.json   # 플러그인 매니페스트
├── agents/          (14개)      # 컴파일된 에이전트
├── skills/          (21개)      # 워크플로우 스킬
├── modules/         (13개)      # 스택별 컨벤션
├── presets/         (2개)       # 프리셋 조합
└── plugins/smith/                 # Smith 에이전트 시스템
```

이 전환 과정에서 몇 가지 정리도 했다:

- **에이전트 16 → 14개 병합**: security-reviewer를 code-reviewer에 mode(quality/security/both)로 통합, tdd-guide를 testgen에 mode(generate/tdd)로 통합
- **5단계 품질 게이트 → thinking-model 내재화**: 4개 파일 수백 줄의 검증 규칙을 5가지 불변 제약으로 압축
- **모듈 13개로 스택 조합**: Framework 3종(Pages Router, App Router, SPA) + Design System 2종(MUI, Ant Design) + State 3종 + Styling 3종 + Testing 2종

App Router를 쓰는 프로젝트면 `react-nextjs-app` 모듈이 로드되고, Pages Router면 `react-nextjs-pages`가 로드된다. 에이전트와 스킬은 동일하다.

## 3. Smith: 에이전트를 컴파일한다

플러그인으로 전환하면서 가장 큰 설계 변경은 Smith(Agent Smith System)의 도입이었다.

Smith는 혼자 만든 게 아니다. 동료인 팩스와 "이 플러그인을 어떻게 하면 더 좋은 방향으로 발전시킬 수 있을까" 고민하던 중에 나온 개념이다. 당시 나는 클린 아키텍처 책을 읽고 있었는데, 인터페이스 추상화와 의존성 역전 원칙이 계속 머릿속에 맴돌았다. 그러다 문득 떠오른 생각 — "에이전트도 인터페이스처럼 STATE(지식)와 ACT(행동)로 분리하면, 레고 블록처럼 조합해서 만들 수 있지 않을까?"

팩스와 이 아이디어를 구체화하면서, 단순히 에이전트를 나열하는 것이 아니라 **소프트웨어 설계 원칙을 에이전트 시스템에 적용**하는 방향으로 발전시켰다. 이후 Codex + Critic 토론(섹션 4에서 다룬다)으로 구조를 교차 검증한 것도 이 맥락에서 자연스럽게 이어진 흐름이다.

Smith는 에이전트를 **STATE(지식)**와 **ACT(행동)**로 분리해서 정의하는 시스템이다.

```
STATE: 이 에이전트가 아는 것 (언어, 프레임워크, 규칙...)
ACT:   이 에이전트가 하는 것 (분석, 구현, 리뷰...)
```

예를 들어 `code-reviewer` 에이전트는:
- STATE: `role.code-reviewer` + `language.typescript` + `framework.react` 체인
- ACT: `quality.code-review` + `quality.security-scan` 체인

이걸 조합하면 에이전트의 전체 프롬프트가 만들어진다.

<!-- 📸 [Smith 조합 다이어그램] STATE 블록(role, language, framework)과 ACT 블록(analysis, dev, quality)이
     레고처럼 맞물려 하나의 에이전트 .md 파일이 되는 흐름. 생성 이미지 적합.
     code-reviewer를 예시로 사용하면 구체적이다. -->

### 런타임 해석의 문제

처음에 Smith는 **런타임 해석** 방식이었다. 매 세션마다:

1. `session.sh`가 실행되어 심링크를 전환
2. `smith-rules.md` 164줄이 세션에 주입되어 STATE/ACT 상속 규칙을 매번 해석
3. 에이전트를 spawn할 때마다 STATE 체인을 재귀적으로 재계산

토큰 낭비, 세션 시작 지연, 에이전트 이중 관리(agents-default/ 14파일 + _agents/ 14파일 = 28파일). 특히 Smith 인스턴스의 `permissionMode: bypassPermissions`가 심링크 경로에서 제대로 적용되지 않는 치명적 버그가 있었다. 분석 전용 에이전트가 코드를 수정하는 상황이 발생할 수 있다는 뜻이었다.

### TypeScript를 매번 해석하는 사람은 없다

해결책은 단순했다.

> TypeScript 코드를 매번 런타임에 해석하지 않고 빌드타임에 .js로 컴파일하는 것처럼,
> Smith 인스턴스를 매번 런타임에 해석하지 않고 `/smith-build`로 정적 .md 파일로 컴파일한다.

```
Smith 인스턴스 (_agents/*.md) → /smith-build → 플랫 .md (agents/)
```

<!-- 📸 [빌드 파이프라인 흐름도] 왼쪽에 _agents/ 소스 파일들 → 중간에 /smith-build 7단계 파이프라인 →
     오른쪽에 agents/ 출력 파일들. TypeScript→tsc→.js 비유와 병렬로 보여주면 이해가 빠르다.
     생성 이미지 권장. 또는 실제 `/smith-build` 실행 터미널 캡처도 좋다. -->

`/smith-build`는 7단계 파이프라인으로 동작한다:
1. Smith 인스턴스 파싱
2. STATE 체인 재귀 해석 + union
3. ACT 체인 해석 + 워크플로우 추출
4. Priority 시맨틱 적용
5. 네이티브 frontmatter 생성
6. 정적 .md 파일 출력
7. 빌드 매니페스트 생성

결과:

| Before (런타임 해석) | After (빌드타임 컴파일) |
|---|---|
| 매 세션: smith-rules.md 164줄 주입 | 0줄 (규칙이 에이전트 파일에 이미 컴파일됨) |
| 매 spawn: STATE 체인 재계산 | 0회 (이미 합산되어 본문에 포함) |
| 28파일 이중 관리 | 14파일 단방향 (소스 → 출력) |
| permissionMode 무시 | 네이티브 frontmatter에 직접 명시 |
| SessionStart hook 실행 | hook 없음. 에이전트가 바로 로드됨 |

런타임 인프라(session.sh, agents-default/ 14파일, smith-activate 스킬)를 전부 삭제했다. 이건 "복잡성을 제거"한 게 아니라 "복잡성을 빌드타임으로 옮긴" 것이다. 컴파일된 에이전트 파일은 단순하지만, 그 단순함은 Smith의 STATE/ACT 체인이 미리 계산된 결과다.

## 4. Codex + Critic 3자 토론으로 구조를 검증하다

code-forge가 어느 정도 형태를 갖추자, 한 걸음 물러서서 "이게 정말 잘 설계된 건가?"를 검증하고 싶었다. 혼자 만든 시스템은 맹점이 생기기 마련이니까.

그래서 **Agent Teams으로 교차 모델 토론**을 진행했다. 참여자는 셋:

- **Codex** (OpenAI) — 외부 시각으로 구조 평가
- **Critic** (Claude Opus) — 내부 품질 감시자
- **나** — 최종 판단

<!-- 📸 [Agent Teams 토론 실제 화면] Claude Code 터미널에서 Codex / Critic이 번갈아 응답하는 실제 세션.
     "CONDITIONAL REJECT" 텍스트가 들어간 부분이 특히 임팩트 있다.
     실제 캡처가 가장 좋다. 어려우면 대화 흐름 다이어그램으로 대체. -->

Codex에게 프로젝트 구조를 보여주고 리팩토링 제안을 받았다. 돌아온 평가는 **CONDITIONAL REJECT**:

1. `read-parallelization.md`와 `parallel-execution.md`가 사실상 같은 내용
2. `forbidden-patterns.md`의 코드 품질 규칙이 `coding-standards.md`와 중복
3. 에이전트 14개의 `@참조` 매트릭스가 역할에 비해 과도

Critic이 이걸 받아서 교차 검증했다. 흥미로운 건 Critic의 반응이었다:

> "1번과 2번은 동의한다. 하지만 3번의 @참조 축소는 주의가 필요하다. 무조건 줄이는 게 답이 아니다. 각 에이전트가 참조하는 컨텍스트는 역할에 맞게 선별된 것이지, 무작위로 붙인 게 아니다."

이 피드백이 중요했다. **토큰 효율과 품질 가드 사이의 균형**이 핵심이었다. 분석 에이전트가 `coding-standards.md`를 참조하는 건 토큰 낭비가 아니라 리뷰 품질을 위한 의도적 선택일 수 있다 — 하지만 그 의도가 역할별로 정확해야 한다.

결과적으로:
- `read-parallelization.md`를 `parallel-execution.md`에 통합 (중복 제거)
- `forbidden-patterns.md`의 코드 품질 규칙을 `coding-standards.md`로 흡수
- @참조 매트릭스는 **역할별로 재정리** (무작정 축소가 아니라, 에이전트 권한 단계에 따라 필요한 참조만 부여)

Codex의 재평가: **CONDITIONAL OKAY**. 방향은 맞다.

이 과정에서 배운 건, **다른 모델과의 토론이 맹점을 찾아준다**는 것이다. 같은 Claude 모델끼리는 비슷한 편향을 공유하지만, GPT 계열은 다른 각도에서 문제를 지적한다.

## 5. 4단계 에이전트 분류와 @참조 매트릭스

Codex + Critic 토론의 가장 중요한 결과물은 에이전트 권한 체계의 재설계다. 기존에는 단순히 "READ-ONLY vs READ-WRITE"로 나눴는데, 실제로는 더 세밀한 분류가 필요했다.

에이전트가 사용할 수 있는 도구(tools)를 기준으로 4단계를 정의했다:

| 분류 | 에이전트 | 허용 도구 |
|------|---------|----------|
| **READ-ONLY** | analyst, architect, refactor-advisor, vision | Read, Grep, Glob만 |
| **SHELL-ACCESS** | scout, code-reviewer, git-operator, researcher | + Bash (Write/Edit 없음) |
| **EDIT-ONLY** | lint-fixer, build-fixer | + Edit (Write 없음) |
| **READ-WRITE-FULL** | implementor, deep-executor, testgen, codex | Write 포함, 모든 도구 |

<!-- 📸 [4단계 권한 다이어그램] 피라미드 또는 계단 구조로.
     아래: READ-ONLY (가장 많은 에이전트, 가장 제한적).
     위로 갈수록: SHELL-ACCESS → EDIT-ONLY → READ-WRITE-FULL.
     각 단계에 에이전트 이름과 허용/차단 도구 표기. 생성 이미지 권장. -->

READ-ONLY 에이전트는 코드를 절대 수정할 수 없다. SHELL-ACCESS 에이전트는 Bash로 명령을 실행할 수 있지만(scout의 빠른 탐색, code-reviewer의 lint 실행, git-operator의 git 명령, researcher의 외부 도구) 파일을 직접 수정하진 않는다. EDIT-ONLY는 기존 파일을 수정할 수 있지만 새 파일을 생성(Write)하지 않는다. READ-WRITE-FULL만이 모든 도구를 사용한다.

핵심은 이 분류에 따라 **@참조 매트릭스를 최적화**한 것이다. 각 에이전트가 로드하는 컨텍스트 파일을 역할에 맞게 선별한다:

| 에이전트 단계 | parallel-execution | coding-standards |
|-------------|:--:|:--:|
| READ-ONLY | -- | -- |
| SHELL-ACCESS (scout, code-reviewer) | O | -- |
| SHELL-ACCESS (git-operator, researcher) | -- | -- |
| EDIT-ONLY | -- | O |
| READ-WRITE-FULL | O | O |

왜 이렇게 나누는가?

- **READ-ONLY** 에이전트는 코드를 수정하지 않으니 `coding-standards.md`가 필요 없다. 병렬 실행도 분석 결과가 독립적이니 `parallel-execution.md`도 불필요하다.
- **scout, code-reviewer**는 여러 파일을 동시에 탐색/리뷰하므로 `parallel-execution.md`를 참조하지만, 코드를 수정하지 않으니 `coding-standards.md`는 필요 없다.
- **git-operator, researcher**는 각각 순차적 Git 작업과 외부 조사에 특화되어 있어 둘 다 불필요하다.
- **EDIT-ONLY** 에이전트(lint-fixer, build-fixer)는 코드를 수정하니 `coding-standards.md`를 반드시 참조해야 하지만, 단일 오류를 수정하는 것이므로 병렬 실행 가이드는 필요 없다.
- **READ-WRITE-FULL** 에이전트는 복잡한 구현을 수행하므로 둘 다 필요하다.

이렇게 역할별로 참조를 최적화하면 토큰을 줄이면서도 품질 가드를 유지할 수 있다.

## 6. 말랑한 하네스를 만든다는 것

에이전트 시스템을 만들면서 느낀 건, **프롬프트만큼 하네스가 중요하다**는 것이다. 하네스란 에이전트가 실행되는 환경 — 도구, 훅, 메모리, 컨텍스트 관리 — 을 말한다. 최근 AI 커뮤니티에서 "Context Engineering"이라는 용어로 불리는 것과 같은 맥락이다.

> 좋은 프롬프트를 작성하는 것만으로는 부족하다.
> 에이전트가 **올바른 시점에 올바른 맥락을 가지고** 동작하게 만드는 것이 핵심이다.

클린 아키텍처 2장에서 "소프트웨어는 말랑해야 한다"고 했다. 변경 비용은 변경 범위에 비례해야 하고, 변경의 난이도는 변경의 형태와 무관해야 한다. 이 기준을 에이전트 하네스에 대입하면 명확해진다.

- 에이전트를 하나 추가할 때 다른 에이전트가 안 깨지는가?
- 프롬프트 규칙을 하나 수정할 때 전체 동작이 안 흔들리는가?
- 다른 프로젝트에 가져갈 때 설정만 바꾸면 되는가?

code-forge의 모듈 시스템, Smith의 STATE/ACT 분리, 빌드타임 컴파일 — 이것들은 전부 하네스를 "말랑하게" 유지하기 위한 설계다. 그리고 코드보다 에이전트 하네스가 더 무서운 점이 하나 있다. 프롬프트 간의 의존성은 **암묵적**이라는 거다. 코드는 컴파일러가 의존성을 잡아주기라도 하지만, 프롬프트는 어디서 뭐가 꼬였는지 추적이 안 된다. 에이전트가 갑자기 이상한 결과를 뱉는데 어떤 프롬프트 수정이 원인인지 찾으려면 하나하나 되돌려봐야 한다. 그래서 하네스의 "말랑함"은 코드의 말랑함보다 더 의도적으로 설계해야 한다.

code-forge가 활용하는 하네스 기능들:

### Hooks 시스템

파일을 수정할 때마다 훅이 실행된다. PreToolUse로 위험한 동작을 사전에 막을 수 있고, PostToolUse로 수정 결과를 사후 검증할 수 있다. code-forge에서는 Mac 알림 훅(`/setup-notifier`)으로 에이전트가 승인을 기다릴 때 배너 알림을 보내는 것도 여기에 해당한다.

<!-- 📸 [Mac 알림 배너 캡처] 에이전트가 승인 요청할 때 Mac 상단에 뜨는 실제 알림 배너.
     독자들이 "이런 게 되는구나" 하고 바로 느낄 수 있다. 실제 캡처 권장. -->

### MCP Tool Deferral

에이전트가 사용하지 않을 MCP 도구를 초기 로딩에서 제외한다. code-forge에서 Figma MCP, Atlassian MCP는 필요할 때만 로드된다. 초기 컨텍스트 토큰을 줄이는 데 직접적으로 기여한다.

### Memory Frontmatter

```markdown
---
name: feedback_testing
description: integration tests must use real database
type: feedback
---
```

에이전트가 세션 간에 학습한 내용을 유지한다. code-reviewer가 "이 프로젝트에서는 MSW 대신 실제 DB를 쓴다"를 한번 배우면 다음 세션에서도 기억한다.

<!-- 📸 [메모리 파일 실제 캡처] .claude/projects/.../memory/ 디렉토리 안의 실제 .md 파일.
     프론트매터(type: feedback, description)가 보이는 에디터 또는 터미널 캡처.
     "에이전트가 기억한다"는 개념을 직접 보여주는 가장 효과적인 방법. -->

### Smith 빌드타임 컴파일도 Context Engineering이다

생각해보면, Smith의 런타임 → 빌드타임 전환도 Context Engineering의 한 형태다. 매 세션마다 164줄의 규칙을 해석하게 하는 대신, 빌드타임에 미리 계산해서 에이전트 파일에 내장시킨 것. 에이전트가 "올바른 시점에 올바른 맥락"을 가지도록, 맥락의 준비 시점을 앞당긴 것이다.

이런 하네스 기능들을 조합하면, 프롬프트 엔지니어링만으로는 불가능한 수준의 에이전트 행동 제어가 가능해진다.

## 7. 마무리: 3번 갈아엎고 도착한 곳

<!-- 📸 [버전 진화 타임라인] v0 → v1 → v1.5 → v2 를 가로 타임라인으로.
     각 버전마다 핵심 키워드:
     v0: "직접 배치", v1: "6단계/14에이전트", v1.5: "모듈화/런타임", v2: "4단계/빌드타임".
     커버와 다른 스타일로 생성 권장. -->

| 버전 | 핵심 변화 | 사고 모델 |
|------|-----------|----------|
| v0 | .claude/ 설정 파일에 규칙과 에이전트를 직접 배치 | 없음 |
| v1 | 14개 에이전트, 5-Gate, /start→/done 워크플로우 | 6단계 (READ→REFLECT) |
| v1.5 | 모듈 시스템 도입, Smith 런타임, 플러그인 패키징 | 4단계 (GROUND→ADAPT) |
| v2 | Smith 빌드타임 컴파일, 런타임 인프라 제거 | 4단계 (동일, 1파일 99줄) |

### 사고 모델의 변화가 가장 중요했다

돌아보면, 에이전트 수를 늘리거나 스킬을 추가한 것보다 **사고 모델을 단순화한 것**이 가장 큰 효과를 줬다. 6단계를 4단계로 줄이면서 에이전트가 오히려 더 정확하게 동작했다. 규칙이 줄어들면 에이전트가 "어떤 규칙을 우선할지" 고민하는 시간이 사라지고, 남은 규칙을 충실히 따른다.

### 다른 모델과의 토론은 습관이 될 만하다

Codex의 CONDITIONAL REJECT는 실제로 중복 파일 정리와 @참조 최적화로 이어졌다. 같은 모델 계열끼리는 비슷한 방식으로 생각하기 때문에, 의도적으로 다른 모델을 끌어들이는 게 유용하다.

### "측정 가능한 구조"가 결국 남는다

Smith 빌드타임 컴파일이 "세션당 ~200줄 주입 절감"이라는 숫자를 낼 수 있는 건, Before/After를 비교할 수 있는 구조로 만들었기 때문이다. 아직 부족한 부분이 많지만(게이트 통과율, 모델 라우팅 비용 절감률 등), 측정 가능한 구조를 만들어두면 개선은 시간 문제다.

### Context Engineering이 프롬프트 엔지니어링을 대체한다

프롬프트를 잘 쓰는 것도 중요하지만, 결국 에이전트의 행동을 결정하는 건 "어떤 맥락이 어떤 시점에 주입되는가"다. Smith 빌드타임 컴파일, MCP Tool Deferral, Memory Frontmatter, Hooks — 이것들은 전부 "올바른 맥락을 올바른 시점에 제공하는" 도구들이다.
