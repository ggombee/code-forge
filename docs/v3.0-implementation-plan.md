## 대장간 체계 (컨셉, 코드에 강제하지 않음)

CLAUDE.md 상단에 선언. 디렉토리명은 Claude Code 스펙을 따르되, 문서에서 메타포 사용.

```
Forge (대장간)     = code-forge 전체
Smith (대장장이)   = 에이전트 빌드 시스템 (현재 "Smith"로 명명, 리네이밍 예정)
Smith (작업대)     = 사용자 인터페이스 (CLI, 향후 대시보드)
Whetstone (숫돌)   = /practice 코딩 연습
Quencher (담금질)  = 검증/비평 (code-reviewer, quality)
Bellows (풀무)     = 로깅/관찰 (hooks/ 기반)
Blueprint (설계도) = rules/ (thinking-model, coding-standards)
Tongs (집게)       = skills/
Mold (거푸집)      = modules/, presets/
```

현재 `plugins/smith/`이 에이전트 빌드 시스템인데, 사용자 교정에 따라 이건 **Smith**(대장장이)여야 함.

```

/smith-build → /smith-build
/smith-create-agent → /smith-create-agent
.smith-build-manifest.json → .smith-build-manifest.json
Agent Smith System → Agent Smith System
```

```bash

DROP TABLE | DELETE FROM.*WHERE 1 | --no-verify
dangerouslyDisableSandbox | curl.*\| sh | eval\(
```

### 1-3. lint-fix.sh 구현

```bash
# package.json에서 lint 스크립트 감지 → --fix 실행
```

### 1-4. 사고모델 에이전트 주입

현재 thinking-model은 `alwaysApply: true`라 메인 Claude에만 적용됨.
sub-agent(implementor, deep-executor 등)에는 **참조가 없음**.

수정: EDIT/WRITE 에이전트에 thinking-model @참조 추가:

```markdown
# implementor.md, deep-executor.md, testgen.md, build-fixer.md 등
@${CLAUDE_PLUGIN_ROOT}/rules/thinking-model.md    ← 추가
@${CLAUDE_PLUGIN_ROOT}/rules/coding-standards.md
```

Smith 빌드 파이프라인의 instruction 매핑 테이블도 수정:
```
| EDIT-ONLY       | lint-fixer, build-fixer | coding-standards, thinking-model |
| READ-WRITE-FULL | implementor, deep-executor, testgen | parallel-execution, coding-standards, thinking-model |
```

## Phase 2: 사용자 친화적 설정 (1주)

### 2-1. /setup 개선 — 옵셔널 기능 대화형 설정

현재 `/setup`이 스택만 설정하는데, 플러그인의 옵셔널 기능도 대화형으로 설정:

```
/setup 완료 후 추가 설정:

code-forge의 기능을 선택해 주세요:

1. Smith (에이전트 빌드 시스템)
   프로젝트 전용 에이전트를 만들 수 있습니다.
   → [사용] / [사용 안 함 (기본)]

2. Whetstone (코딩 연습)
   면접/라이브 코딩 시뮬레이션을 할 수 있습니다.
   → [사용 (기본)] / [사용 안 함]

3. Bellows (사용량 로깅)
   어떤 에이전트/스킬을 얼마나 쓰는지 로컬 로그를 남깁니다.
   → [사용] / [사용 안 함 (기본)]

선택하신 기능만 활성화됩니다. 나중에 변경 가능합니다.
```

설정 결과를 `code-forge.local.md`에 저장:

```yaml
plugins:
  smith:
    enabled: false
  whetstone:
    enabled: true
  bellows:
    enabled: false
```

### 2-2. 작업 규모별 자동 분기 (이미 설계됨, 명시화)

thinking-model이 이미 S/M/L 규모 분기를 가지고 있음:

```
S (1파일, 명확) → 즉시 APPLY     → haiku 에이전트도 충분
M (2-5파일)    → 패턴 확인 후    → sonnet
L (5+파일)     → architect 검토   → opus + 병렬 에이전트
```

이 분기가 **간단한 작업에서 과잉을 방지**함. S 규모에서는 scout 병렬 호출이나 Plan 에이전트가 안 뜸. 이 점을 README에 명시하여 "가벼운 작업에서도 무거운가?" 우려를 해소.

### 2-3. /practice에 knowledge/ 추가

```
skills/practice/
├── SKILL.md
└── knowledge/
    ├── react-best-practices.md
    ├── typescript-patterns.md
    ├── performance-checklist.md
    └── common-antipatterns.md
```

면접관이 이 knowledge를 참조하여 "이건 React 베스트 프랙티스 위반입니다" 같은 피드백 가능.

별도 플러그인이 아닌 hooks/ 확장:

```
hooks/

├── lint-fix.sh
```

로그 위치: `~/.code-forge/usage.jsonl`

```jsonl
{"ts":"2026-03-21T10:00:00Z","skill":"start","project":"a1b2c3"}
{"ts":"2026-03-21T10:00:45Z","agent":"implementor","model":"sonnet","success":true}
```
