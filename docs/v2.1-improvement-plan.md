# code-forge v2.1 개선 계획서

> 4라운드 토론 (감사 → 전략 → Codex 통합 → 방향성) 결과 종합
> 12개 에이전트(Claude/Codex/Critic × 4라운드)가 코드베이스를 직접 읽고 검증한 결과

---

## 1. 플러그인의 본질적 목적

code-forge는 **"어떤 레포지토리든, 그 레포에 최적화된 에이전트 환경을 세팅해주는 Claude Code 플러그인"**이다.

핵심 가치:
- `/setup` 한 번으로 프로젝트 스택에 맞는 CLAUDE.md + 모듈 컨벤션 자동 생성
- 14개 전문 에이전트가 그 레포의 맥락/정책/컨벤션을 이해하고 동작
- Smith 빌드타임 컴파일로 토큰 최적화
- 프리셋/모듈 조합으로 다양한 스택 지원
- 플러그인 업데이트 시 자동 동기화

모든 개선 판단은 이 목적에 얼마나 기여하는가를 기준으로 내렸다.

## 즉시 수정 (v2.1.x)

### 수정 1: hooks.json에서 Stop/SubagentStart/SubagentStop 블록 제거

**문제**: hooks.json에 Stop 훅이 정의되어 있다:
```json
"Stop": [{ "hooks": [{ "type": "command", "command": "cd ${CLAUDE_PLUGIN_ROOT} && yarn lint && yarn tsc --noEmit" }] }]
```
이 명령은 **code-forge 플러그인 디렉토리**에서 `yarn lint`와 `yarn tsc`를 실행한다. 그런데 code-forge는 순수 마크다운/셸 스크립트 프로젝트이므로 `package.json`, `node_modules`, `tsconfig.json`이 없다. 따라서 `yarn lint`는 "package.json not found"로 **항상 실패**한다.

Stop 훅은 Claude Code가 **매 응답 완료 시** 자동 실행하므로, 사용자가 code-forge를 설치한 순간부터 매 대화마다 에러가 발생한다. 사용자가 보는 것은 의미 없는 yarn 에러 메시지다.

SubagentStart/SubagentStop도 `metrics.sh`를 참조하는데, 이 파일이 존재하지 않아 서브에이전트 시작/종료 시마다 에러가 발생한다.

**결론**: 동작하지 않는 훅을 선언해두면 사용자 경험만 해친다. 제거가 맞다.

### 수정 2: lint-fix.sh, guard.sh stub 생성

**문제**: hooks.json이 3개 스크립트를 참조하지만, hooks/ 디렉토리에는 `session-init.sh`와 `notify.sh`만 존재한다:

| hooks.json에서 참조 | hooks/에 존재 | 실행 시점 |
|-------------------|-------------|----------|
| `lint-fix.sh` | **없음** | 매 Edit/Write 후 (PostToolUse) |
| `guard.sh` | **없음** | 매 Bash 실행 전 (PreToolUse) |
| `metrics.sh` | **없음** | 서브에이전트 시작/종료 시 |

Claude Code는 훅 스크립트가 없으면 "command not found" 에러를 반환한다. PostToolUse 훅은 에이전트가 파일을 수정(Edit/Write)할 때마다 실행되므로, 가장 빈번하게 에러가 발생하는 지점이다. PreToolUse는 Bash 명령 실행 전마다 발생한다.

**왜 전체 구현이 아니라 stub(exit 0)인가**: lint-fix.sh의 원래 의도는 "Edit/Write 후 자동 eslint --fix"이고, guard.sh는 "위험 명령 차단"이다. 이것들을 제대로 구현하려면 프로젝트별 lint 명령 감지, 위험 명령 패턴 매칭 등이 필요하다. 이것은 v2.2에서 하고, 지금은 에러를 멈추는 것이 우선이다. `exit 0` stub은 훅이 성공적으로 종료되도록 하여 에러를 제거한다.

**결론**: 존재하지 않는 스크립트 참조로 인한 런타임 에러를 즉시 제거. v2.2에서 실제 기능 구현.

### 수정 3: commands/ 디렉토리 삭제

**문제**: `commands/start.md`(20줄)와 `commands/done.md`(21줄)이 `skills/start/SKILL.md`(179줄), `skills/done/SKILL.md`(244줄)과 중복된다.

더 심각한 문제는 **내용이 잘못되어 있다**는 것이다:
- `commands/start.md` 12번째 줄: `@${CLAUDE_PLUGIN_ROOT}/skills/bug-fix/SKILL.md` 참조 — `/start` 스킬인데 왜 `/bug-fix`를 참조하는가? 잘못된 크로스레퍼런스.
- `commands/done.md` 12번째 줄: 동일하게 `skills/bug-fix/SKILL.md` 참조 — `/done`도 `/bug-fix`를 참조. 복사 붙여넣기 실수.
- `commands/start.md` 17번째 줄: `explore 에이전트 활용` — 이 에이전트는 v2.0에서 `scout`으로 이름이 바뀌었다. 구식 정보.

Claude Code는 `commands/`와 `skills/`를 모두 슬래시 커맨드로 인식할 수 있어, 사용자가 `/start`를 실행할 때 어느 것이 로드되는지 불명확하다. skills/에 완전하고 정확한 버전이 있으므로 commands/는 불필요하다.

**결론**: 정보가 부정확한 중복 파일. 하나의 정보원(skills/)만 유지하는 것이 유지보수에 유리하다.

### 수정 4: 실행 경로 에이전트명 치환

**문제**: v2.0에서 에이전트 이름이 변경되었다:
- `explore` → `scout` (Claude Code 빌트인 Explore와 이름 충돌 방지)
- `implementation-executor` → `implementor` (간결화)

컴파일된 에이전트 파일은 `agents/scout.md`, `agents/implementor.md`로 정확히 존재한다. 그러나 **스킬과 가이드 문서에서 구식 이름을 계속 사용**하고 있다:

- `skills/start/SKILL.md` 109-111행: `subagent_type='explore'`로 에이전트 호출 지시
- `skills/bug-fix/SKILL.md` 88행: `explore` 참조
- `skills/refactor/SKILL.md` 53-67행: `implementation-executor` 참조
- `instructions/multi-agent/agent-roster.md`: `explore`, `implementation-executor`로 테이블 기재
- `instructions/multi-agent/execution-patterns.md`: 구식 이름으로 예시 코드

**왜 이것이 실제 문제인가**: Claude Code의 `Agent` 도구는 `subagent_type` 파라미터로 `agents/` 디렉토리에서 파일을 찾는다. `subagent_type='explore'`를 지정하면 `agents/explore.md`를 찾는데, 이 파일은 없다 (scout.md로 바뀜). 대신 Claude Code **빌트인 Explore 서브에이전트**(코드베이스 탐색용 경량 에이전트)가 실행될 수 있다 — 이것은 code-forge의 scout 에이전트가 아니라 완전히 다른 것이다.

`implementation-executor`는 빌트인에도 없으므로 아예 라우팅이 실패한다.

**왜 43개 전체가 아니라 5-8개만 치환하는가**: 스킬의 Task() 호출부와 가이드의 예시 코드만 **실제 에이전트 라우팅에 영향**을 준다. 나머지는 설명 텍스트("scout 에이전트는 탐색을 담당합니다" 같은)로, 읽는 사람의 이해에는 영향이 있지만 코드 실행에는 영향이 없다. 비용 대비 효과를 고려해 실행 경로만 우선 수정한다.

**결론**: 에이전트 라우팅이 깨지는 실행 경로를 수정. 문서 설명은 권장 수정(v2.2).

### 수정 5: Codex 에이전트 제거

**문제의 본질**: code-forge의 본질적 목적은 "에이전트들이 그 레포의 맥락/정책/컨벤션을 **완전히** 이해하고 동작"하는 것이다. codex 에이전트를 통해 Codex(OpenAI)를 호출하면, Codex는 다음을 모른다:
- CLAUDE.md에 정의된 프로젝트 스택 정보
- modules/의 상세 컨벤션 (디자인 토큰, Emotion 패턴, Jotai 쿼리 키 등)
- rules/의 코딩 표준과 금지 패턴
- instructions/의 작업 절차

`@${CLAUDE_PLUGIN_ROOT}/` 참조는 Codex에서 해석되지 않으므로, code-forge의 모듈 시스템 전체가 Codex에게 전달되지 않는다. 이것은 "맥락을 완전히 이해"라는 약속과 정면으로 모순된다.

**실사용률 분석**: codex 에이전트는 3중 opt-in 장벽이 있다:
1. OpenAI 계정 (Pro/Team/Enterprise 플랜 필요)
2. codex CLI 설치 + `codex login` OAuth 인증
3. `.mcp.json`에 MCP 서버 등록 + Claude Code 재시작

이 3개를 모두 충족하는 사용자만 codex 에이전트를 활성화할 수 있다. 대부분의 환경에서 codex 에이전트는 존재하지만 호출되지 않는 상태다.

**제거해도 기능 공백이 없는 이유**: Codex가 담당하던 역할과 대체 에이전트:
| Codex 역할 | 대체 에이전트 |
|-----------|------------|
| 꼼꼼한 구현 | `implementor` (sonnet) — 계획 기반 즉시 구현 |
| 코드 리뷰 더블체크 | `code-reviewer` (sonnet) — 품질+보안 리뷰 |
| Team Lead | `deep-executor` (sonnet) — Ralph Loop 자율 구현 |
| 엣지케이스 검증 | `testgen` (sonnet) — BDD 시나리오 기반 테스트 생성 |

**유령 모듈 참조 버그**: `/setup` SKILL.md의 모듈 매핑 테이블에 `modules/advanced/codex/SKILL.md`가 참조되어 있지만, `modules/advanced/` 디렉토리 자체가 존재하지 않는다. profile.json에 `"advanced": ["codex"]`를 넣으면 CLAUDE.md에 깨진 @참조가 생성되는 버그다. Codex를 제거하면 이 유령 참조도 같이 정리된다.

**제거 대상 파일** (정확히 3개):
- `agents/codex.md` (270줄, 컴파일된 에이전트)
- `skills/codex/SKILL.md` (192줄, 스킬)
- `plugins/smith/agents/_agents/codex.md` (Smith 인스턴스 소스)

나머지 13개 에이전트와 12개 스킬 중 **어느 것도 codex를 참조하거나 의존하지 않는다.**

**결론**: 본질적 목적(맥락 완전 이해)과 모순, 실사용률 극저, 기능 대체 가능, 유령 참조 유발. 제거가 합리적.

### 수정 6: 유령 모듈 참조 정리

**문제**: `/setup` SKILL.md의 모듈 매핑 테이블(115-132행)에 `advanced` 카테고리가 있다:
```
| advanced | codex  | ${CLAUDE_PLUGIN_ROOT}/modules/advanced/codex/SKILL.md |
| advanced | figma  | ${CLAUDE_PLUGIN_ROOT}/modules/advanced/figma/SKILL.md |
```

그런데 `modules/advanced/` 디렉토리 자체가 없다. `modules/` 하위에는 `frameworks/`, `design-systems/`, `state/`, `styling/`, `testing/` 5개 카테고리만 존재한다.

사용자가 profile.json에 `"advanced": ["codex", "figma"]`를 넣으면, `/setup`이 CLAUDE.md를 생성할 때 존재하지 않는 파일의 @참조가 포함된다. Claude Code가 이 @참조를 해석하려 하면 "파일 없음" 에러가 발생하거나, 모듈 로딩이 실패한다.

에러 처리 섹션에 "모듈 SKILL.md 없음 → 경고 출력 후 해당 모듈 참조 제외하고 계속 진행"이 있긴 하지만, 애초에 매핑 테이블에 존재하지 않는 파일을 나열하는 것 자체가 문제다.

**결론**: 존재하지 않는 모듈 매핑을 제거하거나, 실제 파일을 생성. Codex를 제거하므로 codex 매핑은 삭제. figma는 `/figma-to-code` 스킬이 별도로 있으므로 모듈화 여부를 판단 후 처리.

## 안 하는 것 (근거 포함)

### lint-fixer의 Grep/Glob 차단 해제 — 불필요

**Proposer 주장**: lint-fixer가 Grep/Glob 없이는 오류 컨텍스트(타입 정의 위치 등)를 분석할 수 없다.

**Codex 반론 (채택됨)**: lint-fixer의 실제 워크플로우를 읽으면, Step 1에서 `Bash`로 `yarn lint`/`yarn build` 실행 → 출력에서 파일경로:라인 확인 → `Read`로 해당 파일 읽기 → `Edit`로 수정. 이 흐름에서 Grep/Glob는 필요하지 않다.

오히려 Grep/Glob를 허용하면 lint-fixer가 **lint 수정 범위를 넘어 전체 코드베이스를 스캔**하게 될 수 있다. lint-fixer는 haiku 모델(빠르고 저렴)을 사용하므로, 도구 제한은 "최소 변경 원칙"을 강제하는 의도적 설계다.

### 분석 에이전트 Bash 차단 / permissionMode 변경 — 불필요

**Proposer 주장**: scout, researcher, code-reviewer가 Bash를 통해 `sed -i`, `rm` 등으로 파일을 수정할 수 있다. permissionMode를 bypassPermissions에서 default로 변경해야 안전하다.

**Codex 반론 (채택됨)**:
1. 이 에이전트들의 Bash 사용 목적은 `git log`, `git blame`, `git diff` 등 **히스토리 조회**다. Bash를 차단하면 이 핵심 기능이 불가능해진다.
2. 각 에이전트의 `<forbidden>` 섹션에서 "코드 수정 금지"가 명시적으로 선언되어 있다. 이것은 프롬프트 수준의 방어선이다.
3. `bypassPermissions`는 "허용된 도구를 승인 없이 실행"이라는 의미이지, "차단된 도구를 풀어주는 것"이 아니다. Write/Edit가 disallowedTools에 있으므로 bypassPermissions와 무관하게 차단된다.
4. 서브에이전트는 사용자와 직접 상호작용할 수 없어, 승인 대기 시 **데드락**이 발생한다. bypassPermissions는 병렬 실행에 필수다.

### 에이전트 선택 결정 트리 추가 — 불필요

**Proposer 주장**: 14개 에이전트 중 어떤 것을 써야 하는지 결정 트리가 필요하다.

**Codex 반론 (채택됨)**: 이미 존재한다.
- `instructions/multi-agent/agent-roster.md`: 전체 에이전트 목록, 분류, 용도, 예시
- `instructions/multi-agent/coordination-guide.md`: 모델 라우팅 기준, 에이전트별 용도 테이블
- `instructions/agent-patterns/model-routing.md`: 에이전트별 기본/상향 모델 테이블

추가 문서는 중복이다.

### 스킬 에러 처리 추가 — 과잉 엔지니어링

**Proposer 주장**: 13개 스킬 중 대부분에 실패 시나리오 처리가 없다.

**Codex 반론 (채택됨)**: code-forge의 스킬은 LLM 프롬프트 문서다. 일반 코드의 try/catch와 다르다. Claude는 에러를 자연어로 인지하고 보고하며, 구조적 에러 핸들링은 프롬프트 복잡성만 증가시킨다. 실제로 `/start` 스킬에는 이미 "Figma MCP 실패 시" 조건 분기가 자연어로 기술되어 있다 — 이것이 LLM 프롬프트에서의 올바른 에러 처리 방식이다.

### AGENTS.md 자동 생성 — 불필요

**이전 토론에서 검토됨**: Codex의 `project_doc_fallback_filenames = ["CLAUDE.md"]`를 설정하면 Codex가 CLAUDE.md를 읽을 수 있다. 그러나 CLAUDE.md 내 `@${CLAUDE_PLUGIN_ROOT}/modules/...` 참조가 Codex에서 해석되지 않아, 스택 요약(Framework: Next.js 등) 한 줄만 얻고 상세 컨벤션은 도달 불가. AGENTS.md를 별도 생성해도 같은 문제 — @참조 없이 모든 컨벤션을 평문으로 풀면 수백 줄이 되어 유지보수 이중화.

Codex 에이전트를 제거하므로 이 문제 자체가 해소된다.

### Smith 듀얼 빌드 — 불필요

Codex config.toml의 `[agents.역할]`과 Smith 컴파일 결과(.md)의 기능 동등성이 없다. Smith의 STATE+ACT 조합, 도구 제한(tools/disallowedTools), 모델 라우팅, Must/Never/Should 규칙 합산 등을 config.toml로 표현할 방법이 없다.

### 감지순서 반전 (CLI 먼저) — 불필요

`agents/codex.md` 112-117행의 "모드 자동 선택 기준"이 이미 요청 유형별 분기를 정의한다:
- 단일 질의 → CLI Headless
- 멀티턴 → MCP

감지순서(MCP 먼저 체크)와 모드 선택(요청 유형으로 결정)은 별개 로직이다. 현재 구현이 올바르다. 다만, Codex를 제거하므로 이 논점 자체가 해소된다.

## v2.2 로드맵 (단기)

### 백엔드 모듈 추가

**문제**: "어떤 레포지토리든"이라는 약속과 현실의 괴리. 현재 modules/의 14개 모듈은 **전부 React/Next.js 프론트엔드**다:
- frameworks: react-nextjs-pages, react-nextjs-app, react-spa
- design-systems: mui, ant-design
- state: jotai-tanstack, zustand-tanstack, redux-rtk
- styling: emotion, tailwind, styled-components
- testing: jest, vitest

Smith의 STATE class에는 이미 Django, Fastify, Python, Vue 등이 포함되어 있다(`plugins/smith/agents/state/framework/`). 그런데 모듈 시스템에는 대응하는 컨벤션이 없어서, Smith가 "나는 Django를 안다"고 선언해도 실제 Django 컨벤션(URL 패턴, serializer 규칙, migration 정책 등)은 제공되지 않는다.

### /setup 자동감지

**문제**: `/setup`이 프로젝트 스택을 자동 감지하지 않고 대화형 선택에 의존한다. 그런데 `/smith-create-agent`는 Step 2에서 package.json, tsconfig.json, 소스 구조를 자동 분석한다. 같은 플러그인 안에서 하나는 자동이고 하나는 수동인 것은 불일치다.

`next` in dependencies → react-nextjs, `@emotion/styled` → emotion, `jest` in devDeps → jest 같은 매핑은 자명하다.

### lint-fix.sh / guard.sh 실구현

**가치**: lint-fix.sh = Edit/Write 후 자동 `eslint --fix`. 에이전트가 코드를 수정할 때마다 자동으로 린트가 적용되어 코드 품질이 일관되게 유지된다. guard.sh = `rm -rf /`, `git push --force origin master`, `DROP TABLE` 같은 위험 명령을 Bash 실행 전에 차단. 안전망 역할.

## v3.0 로드맵 (중기)

### 에이전트-모듈 직접 참조 검토

**발견된 구멍**: 컴파일된 에이전트(agents/*.md) 14개 중 **어느 것도 modules/를 직접 참조하지 않는다.** 에이전트들이 참조하는 것은 instructions/와 rules/뿐이다.

모듈 컨벤션은 CLAUDE.md의 @참조를 통해 **메인 컨텍스트에만** 로드된다. 서브에이전트(Agent로 spawn된 에이전트)가 CLAUDE.md를 상속받고, 그 안의 @참조를 재귀적으로 해석하는지는 Claude Code의 구현에 의존한다. 만약 재귀 해석이 안 되면, 서브에이전트는 모듈 컨벤션을 모르는 상태에서 동작하게 된다.

핵심 구현 에이전트(implementor, deep-executor, testgen)에 모듈 @참조를 직접 주입하는 옵션을 검토할 필요가 있다.

### Smith STATE ↔ modules 경계 문서화

React 지식이 두 곳에 분산되어 있다:
- `plugins/smith/agents/state/framework/react.md`: "나는 React를 안다" (에이전트 페르소나의 지식 배경)
- `modules/frameworks/react-nextjs-pages/SKILL.md`: "이 프로젝트에서 React를 이렇게 쓴다" (프로젝트 컨벤션)

역할이 다르므로 통합보다는 경계를 명확히 문서화하는 것이 적절하다.

### 스킬 레지스트리 opt-in 통합

`~/Desktop/ai/registry-service/` (Django REST API)와의 연동:
- `/skill-publish`: 로컬 스킬을 레지스트리에 등록
- `/skill-install`: 레지스트리에서 스킬 검색 → 다운로드 → 로컬 설치
- 원칙: "레지스트리는 배포 채널, 런타임 의존 아님" — 설치 후에는 100% 로컬 동작
