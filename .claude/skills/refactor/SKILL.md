# Refactor Skill

> 기존 정책을 보호하면서 코드 구조 개선

---

## 트리거 조건

| 트리거 | 반응 |
|--------|------|
| "리팩토링", "구조 개선" | 스킬 활성화 |
| "중복 제거", "분리" | 스킬 활성화 |
| 코드 리뷰에서 구조 개선 제안 | 분석 시작 |

---

## 핵심 원칙

| 원칙 | 설명 |
|------|------|
| **정책 보호** | 비즈니스 로직 동작 유지 |
| **점진적** | 작은 단위로 단계적 진행 |
| **테스트 우선** | 리팩토링 전 정책 테스트 작성 |
| **최소 변경** | 불필요한 개선 지양 |

---

## 복잡도 판단

| 복잡도 | 기준 | 접근 |
|--------|------|------|
| **LOW** | 변수명 변경, 단순 추출 | 바로 실행 |
| **MEDIUM** | 함수 분리, 파일 구조화 | 계획 후 실행 |
| **HIGH** | 아키텍처 변경, 패턴 도입 | planner 에이전트 활용 |

---

## 리팩토링 판단 기준

| 리팩토링 필요 | 리팩토링 불필요 |
|--------------|----------------|
| 동일 로직 3곳+ 중복 | 1-2곳 유사 코드 |
| 명백한 책임 분리 위반 | 단순 가독성 개선 |
| 테스트 불가능한 구조 | 취향 차이 수준 |
| 500줄+ 파일 | 200줄 이하 파일 |

---

## 워크플로우

### Phase 1: 현황 분석

```typescript
// 병렬 탐색
Task(subagent_type="explore", model="haiku", prompt="리팩토링 대상 코드 분석")
Task(subagent_type="explore", model="haiku", prompt="관련 테스트 현황 파악")
Task(subagent_type="explore", model="haiku", prompt="의존성 파일 목록 추출")
```

**내부 점검 (출력 불필요):**

```
- 현재 구조: {파일/함수 목록}
- 문제점: {중복/복잡도/책임분리}
- 영향 파일: {목록}
- 기존 테스트: {있다/없다}
- 보호할 정책: {목록}
```

---

### Phase 2: 리팩토링 계획

**HIGH 복잡도 시 planner 에이전트 활용:**

```typescript
Task(subagent_type="planner", model="opus", prompt=`
  대상: {리팩토링 대상}
  목표: {개선 목표}
  제약: 기존 정책 유지
`)
```

**계획 포맷:**

```markdown
## 리팩토링 계획

### 목표
{한 문장}

### 변경 계획
| # | 단계 | 파일 | 리스크 |
|---|------|------|--------|
| 1 | {작업} | {파일} | {상/중/하} |

### 정책 보호 항목
- {보호할 정책}

### 롤백 전략
{실패 시 복구 방법}
```

---

### Phase 3: 정책 보호 테스트 작성

**리팩토링 전 반드시 기존 동작 캡처:**

```typescript
describe('리팩토링 전 정책 캡처', () => {
  it('기존 정책이 그대로 동작해야 함', () => {
    // 현재 동작을 "캡처"하는 테스트
  });
});
```

**테스트 대상:** 날짜/기간 계산, 필터 조건, disabled/readonly 조건, 기본값/초기값

---

### Phase 4: 단계적 실행

각 단계 후 검증:
- [ ] lint 통과
- [ ] build 통과
- [ ] 정책 테스트 통과
- [ ] 기존 테스트 통과

```typescript
// 병렬 검증
```

## 금지 패턴

| 금지 | 이유 |
|------|------|
| 테스트 없이 리팩토링 | 정책 변경 감지 불가 |
| 한 번에 큰 변경 | 롤백 어려움 |
| 정책 임의 변경 | 비즈니스 영향 |
| 불필요한 추상화 | 복잡도 증가 |
