# Bug Fix Skill

> 버그를 분석하고 2-3가지 해결 옵션을 제시한 후 선택된 방법으로 수정

---

## 트리거 조건

| 트리거 | 반응 |
|--------|------|
| "버그", "오류", "에러" | 스킬 활성화 |
| 에러 메시지/스택트레이스 제공 | 분석 시작 |
| "동작 안함", "이상해" | 스킬 활성화 |

---

## 복잡도 판단

| 복잡도 | 기준 | 접근 |
|--------|------|------|
| **LOW** | 단일 파일, 명확한 원인 | 원인 명확하면 옵션 없이 즉시 수정 |
| **MEDIUM** | 2-3개 파일, 조사 필요 | 옵션 제시 |
| **HIGH** | 5개+ 파일, 근본 원인 불명 | 상세 분석 후 옵션 제시 |

---

## 워크플로우

### Phase 1: 증상 파악

```typescript
Task(subagent_type="explore", model="haiku", prompt=`
  에러 증상: {에러 메시지}
  관련 키워드로 코드베이스 검색
`)
```

**수집 항목:**
- 에러 메시지 전문
- 발생 조건 (재현 단계)
- 관련 파일/함수
- 최근 변경 내역 (`git log`)

---

### Phase 2: 원인 분석

**내부 점검 (출력 불필요):**

```
- 에러 유형: {타입 | 런타임 | 로직 | 상태 | API}
- 근본 원인: {한 문장}
- 영향 범위: {파일 수, 기능}
- 최근 변경과 관련: {있다/없다}
```

**분석 체크리스트:**
- [ ] 타입 에러인가? (TypeScript)
- [ ] 런타임 에러인가? (null/undefined)
- [ ] 로직 에러인가? (조건문, 계산)
- [ ] 상태 관리 이슈인가?
- [ ] API 응답 이슈인가?

---

### Phase 3: 해결 옵션 제시

**반드시 2-3가지 옵션 제시:**

```markdown
## 버그 분석 결과

### 증상
{증상 설명}

### 원인
{근본 원인 분석}

### 해결 옵션

| # | 방법 | 장점 | 단점 | 영향 범위 |
|---|------|------|------|-----------|
| 1 | {방법1} | {장점} | {단점} | {파일 수} |
| 2 | {방법2} | {장점} | {단점} | {파일 수} |

### 권장
{권장 옵션 번호} - {이유}
```

---

### Phase 4: 수정 및 검증

**검증 체크리스트:**
- [ ] 원래 에러가 해결되었는가?
- [ ] 새로운 에러가 발생하지 않았는가?
- [ ] 기존 동작이 유지되는가?
- [ ] lint/build 통과하는가?

---

## 금지 패턴

| 금지 | 이유 |
|------|------|
| 원인 파악 없이 수정 | 재발 위험 |
| 단일 옵션만 제시 | 선택권 없음 |
| 테스트 삭제로 해결 | 품질 저하 |
| 정책 임의 변경 | 비즈니스 영향 |
