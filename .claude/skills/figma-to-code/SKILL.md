# Figma to Code Skill

> 디자인 시안을 분석하여 프로젝트 컨벤션에 맞는 컴포넌트 코드로 변환

---

## 트리거 조건

| 트리거 | 반응 |
|--------|------|
| 이미지/스크린샷 첨부 + "구현해줘" | 스킬 활성화 |
| "디자인대로", "시안대로" | 스킬 활성화 |
| Figma URL 공유 | 분석 시작 |

---

## 워크플로우

### Phase 1: 디자인 분석

```typescript
// 이미지 분석
Task(subagent_type="vision", model="sonnet", prompt=`
  첨부된 디자인 시안을 분석:
  - 컴포넌트 구조 (레이아웃, 중첩)
  - 색상, 타이포그래피, 간격
  - 상호작용 요소 (버튼, 입력, 토글)
  - 반응형 힌트 (있다면)
`)

// 기존 컴포넌트 패턴 탐색 (병렬)
Task(subagent_type="explore", model="haiku", prompt=`
  기존 UI 컴포넌트 패턴 분석:
  - 공통 컴포넌트 (Button, Input, Card 등)
  - 스타일링 패턴 (프로젝트 스타일링 도구)
  - 레이아웃 패턴
`)
```

**내부 점검 (출력 불필요):**

```
- 컴포넌트 수: {개수}
- 재사용 가능 기존 컴포넌트: {목록}
- 새로 만들어야 할 컴포넌트: {목록}
- 예상 파일 수: {개수}
```

---

### Phase 2: 컴포넌트 설계

**분해 원칙:**
- 재사용 가능한 단위로 분리
- 기존 컴포넌트 최대 활용
- props 인터페이스 명확히 정의

```markdown
## 컴포넌트 구조

| 컴포넌트 | 유형 | 기존/신규 | 위치 |
|---------|------|----------|------|
| {이름} | {레이아웃/UI/기능} | {기존 활용/신규} | {경로} |
```

---

### Phase 3: 구현

```typescript
Task(subagent_type="designer", model="sonnet", prompt=`
  디자인 시안 기반 컴포넌트 구현:
  - 분석 결과: {Phase 1 결과}
  - 기존 패턴: {Phase 1 탐색 결과}
  - 프로젝트 스타일링 도구 사용 (package.json에서 감지)
  - 반응형 + 다크모드
  - 접근성 (WCAG 2.2 AA)
`)
```

---

### Phase 4: 검증

- [ ] 시안과 시각적으로 일치하는가?
- [ ] 반응형 동작하는가?
- [ ] 다크모드 지원하는가?
- [ ] 기존 컴포넌트 패턴과 일관적인가?
- [ ] lint/type 통과하는가?

## 금지 패턴

| 금지 | 이유 |
|------|------|
| 하드코딩 색상/크기 | 테마 일관성 파괴 |
| inline style 사용 | 프로젝트 스타일링 컨벤션 위반 |
| 기존 컴포넌트 무시 | 중복 코드 |
| px 고정 폰트 사이즈 | 접근성 위반 |
